# 基于大语言模型的智能合约重入漏洞检测

> 消融实验驱动的迭代优化流水线 · 深度求索对话模型 · 代码切片 · 提示词工程

---

## 引言与背景

### 为什么要研究重入漏洞

重入漏洞是智能合约最经典、最具破坏性的安全问题之一，核心风险是"外部调用后控制权被夺回"。合约在执行外部调用时如果尚未完成状态更新，攻击者即可通过回调再次进入同一逻辑，后果通常为余额重复提取、权限状态错乱、领取次数被绕过，最终造成链上资产损失。

传统静态分析工具擅长寻找"表面模式"，但面对跨函数、跨合约、修饰器隐藏调用时常常漏掉真正的攻击路径。大语言模型的价值在于能读懂更长的上下文、整合多文件信息，并用自然语言推理出"调用链 + 状态变化顺序"。

### 重入漏洞的典型类型

1. **单函数重入**：攻击者在同一函数里通过外部回调再次进入原函数，例如经典的提款模式——先外部调用，后更新余额。
2. **跨函数重入**：攻击者回调进入同一合约的另一公共函数，这些函数共享余额、锁变量、白名单或计数器等状态。
3. **跨合约重入**：主合约调用依赖合约或外部接口后，被攻击者引导回到主合约，形成"主合约—外部合约—主合约"的循环。
4. **修饰器重入**：漏洞不在函数体内，而藏在修饰器中；函数主体尚未开始执行，控制权已被外部调用交出。

### 常见攻击手段

- **回退/接收回调**：合约通过 call、send、transfer 向攻击者地址转账，攻击者在回调中再次调用受害函数。
- **递归外部调用**：受害合约在一个函数中多次触发外部合约接口，攻击者利用第一次外部调用构造第二次进入机会。
- **共享状态绕过**：攻击者先触发某个函数修改中间状态，再回调另一个函数绕过原先的检查条件。
- **代币钩子/接口钩子**：在增强代币标准（ERC777）、非同质化代币（ERC721）等场景中，转账本身会触发钩子函数，形成天然回调点。
- **修饰器前置调用**：在修饰器内部先做外部交互，再进入函数主体，攻击者获得"未完成初始化的窗口期"。

最稳妥的防护是检查—生效—交互（Checks-Effects-Interactions）原则：先检查条件，再修改状态，最后做外部交互；必要时再叠加重入锁。

### 文献综述与研究空白

传统方法主要依赖人工审计、规则匹配和静态分析工具，例如 Slither 能快速定位可疑外部调用点。这些方法对"显式模式"有效，但面对跨函数、跨合约、多文件依赖时，往往需要审计者手工补齐上下文。纯大语言模型方法能读长代码、做语义归纳，但容易受到长上下文噪声、提示词设计和幻觉问题的影响。

**研究空白**：如何让大语言模型在"长代码 + 多文件依赖 + 重入语义"场景中，既保持可复现，又能做可解释的判断。

**本研究的切入点**：把静态分析、代码裁剪和提示词工程组合成一个可重复的实验流水线，再用消融实验逐项验证每个增强模块是否真的带来收益。

### 研究目的与创新点

- 建立一个可重复的重入漏洞检测流程：数据整理、预处理、提示词组装、模型预测、结果评估全链路落盘。
- 将深度求索对话模型作为大语言模型基线，比较原始代码、裁剪代码、静态分析摘要、思维链推理、多合约上下文等方案。
- 用统一的结构化输出格式记录漏洞判定、漏洞类型、漏洞函数、攻击路径、置信度和推理依据。
- 通过消融分析回答：哪些上下文真的有帮助，哪些反而会稀释模型判断。

## 研究方法

### 总体设计思路

核心设计哲学：**不是训练一个模型，而是构建一个可解释、可迭代、可消融的检测系统**。

- **端到端 vs 流水线**：纯大语言模型端到端黑盒无法定位"哪一步起作用"，必须拆解为独立模块逐一验证。
- **消融实验驱动的迭代**：每一轮实验暴露问题，设计针对性优化，再实验验证，形成闭环。
- **可复现性优先**：所有中间产物落盘，重复三次报告均值与标准差。

大语言模型能力有限（幻觉、上下文窗口、对 Solidity 语义理解不全），所以不追求全自动，而是用静态分析、规则裁剪与提示词工程补齐短板。

### 实验迭代路线

```
第 1 轮：消融实验（五个方案 × 重复三次）
  → 发现：误报率过高（0.458），固定变体 100% 误判

第 2 轮：切片 v1 + 守卫注入（四项规则优化）
  → 发现：误报率 0.375，切片剥离安全上下文

第 3 轮：提示词规则注入（显式教授 nonReentrant 语义）
  → 最终：误报率 0.250，固定变体准确率 0.750
```

### 关键设计决策与实验协议

| 决策 | 理由 |
|---|---|
| 重复三次 | 单次运行具有随机性，三次报告均值与标准差更可信 |
| 温度等于零（贪婪解码） | 消除采样随机性，确保不同方案之间差异来自输入而非温度 |
| 五个消融方案 | 从简单到复杂逐层叠加，每层验证一个假设 |
| 三百秒超时 | 深度求索接口偶尔超时，三百秒覆盖百分之九十八请求 |
| 选用深度求索对话模型 | 性价比最优，兼容接口，且其 Solidity 语义理解能力恰好暴露了当前大语言模型的典型局限 |

### 数据集与清洗

**四个数据来源**：

1. 序列编码器样本集：八个配对样本，四类场景各含不安全版与修复版。
2. 智慧漏洞精选集：三十一个已验证重入漏洞的链上合约，全为阳性。
3. 额外重入概念验证样本：两个跨函数与跨合约概念验证样本。
4. 跨链重入配对样本集：八个跨链配对样本。

**清洗前的问题**：原始四十九个样本中智慧漏洞精选集存在八个近重复（行级雅卡尔相似度大于百分之九十），多源合并时样本标识碰撞导致目录覆盖，类别严重不平衡（三十七正比四负，比例九点二比一）。

**清洗操作**：百分之九十行级雅卡尔相似度去重（智慧漏洞精选集三十一变二十三），样本标识前缀化消除碰撞，重新加入跨链配对样本。

**清洗后**：四十一份样本，三十三正比八负（比例四点一比一），误报率分母从四升到八，更稳定。

| 序列编码器样本集 | 8 | 4 | 4 |

### 整体方法流程

四阶段流水线：Solidity 合约与依赖文件 → 预处理（静态分析与裁剪匿名化）→ 提示词组装 → 模型结构化预测 → 评估与落盘。预处理阶段保留风险函数、相关修饰器和依赖上下文，并对样本名做匿名化以减少标签泄漏。模型输出采用固定结构化格式，方便批量解析与指标统计。

### 提示词设计与消融配置

**实验动机**：核心问题是"什么样的上下文对大语言模型重入检测最有帮助？是越多越好，还是精准裁剪更重要？"

五个消融方案由简至繁递进：

- **baseline_raw**（完整原文）：纯大语言模型基线——验证是否需要任何预处理。
- **crop_only**（仅裁剪）：仅保留风险代码——验证"去掉无关代码"是否有帮助。
- **crop_slither**（裁剪加摘要）：裁剪后叠加静态分析摘要——验证静态分析信息是否增强判断。
- **crop_slither_cot**（裁剪加摘要加思维链）：加入思维链分步推理——验证复杂提示工程是否有收益。
- **crop_slither_multi**（裁剪加摘要加思维链加多合约）：加入多合约上下文——验证依赖文件是否必要。

消融实验逻辑：从简单到复杂逐层叠加，观察每一步的边际收益。若某层退化，说明该增强可能引入噪声。

**切片 v1 优化方案**：在消融实验基础上，引入四项切片优化规则——外部调用精确化、函数筛选、显著函数前百分之十、贪心填充三千八百字符。

### 提示词规则注入

**设计动机**：切片实验中固定变体被百分百误判。根源在于切片剥离了 ReentrancyGuard 继承链，模型只能看到"危险的调用模式"而看不到"保护措施"。

注入的提示词规则核心内容：
- 如果函数使用了 nonReentrant 或 noReentrant 修饰符，说明已受重入锁保护，不应判定为漏洞。
- nonReentrant 工作原理：执行函数体前设 locked 为 true 上锁，require(!locked) 阻止回调重入。
- 合约继承 ReentrancyGuard 且函数带 nonReentrant 等于安全。

设计选择：保持代码原样、只在提示词中教学。这样代码切片结果不变，所有实验结果可对比，且验证了"仅靠提示词能否补偿模型的语义理解不足"。

### 评价指标与实验协议

三项核心指标：准确率等于真阳加真阴除以总数，误报率等于假阳除以负样本数，漏报率等于假阴除以正样本数。

类别不平衡（三十三正比八负）时，准确率可能虚高。误报率高意味着安全合约被错误判为有漏洞，增加人工复核成本；漏报率高意味着真实漏洞滑过检测，导致链上资产损失风险。需要同时优化两者，而非只看准确率。

**实验协议**：所有对比实验必须使用相同四十一个样本、相同模型、温度等于零、重复三次报告均值与标准差。不同运行标识独立保存，旧实验完全不动。

### 数据集优化：切片引擎 v1

**为什么要做数据集优化？** 前一轮消融发现核心问题：裁剪逻辑缺乏全局视角，未区分"重要合约"与"模板合约"，且固定变体安全上下文丢失。

**四项优化规则**：

1. **外部调用精确化**：只保留重入强相关的低级调用——call、delegatecall、callcode（含 value）、transfer、send，不把 ERC20 代币的 transfer 和 transferFrom 当作强制保留。
2. **函数筛选**：按 function、constructor、fallback、receive 切片；保留代码行数不小于五或含外部调用的函数；被保留函数的修饰器一并保留。
3. **显著函数前百分之十**：统计全局二百三十九个函数，五十四个唯一名，五个显著函数（全局唯一名且长度位于前百分之十）。合约含显著函数或外部调用函数才保留。
4. **输入块构造**：清理注释、import、pragma，外部调用函数优先，按长度贪心填充至字符上限三千八百（第 99 百分位数）。

**压缩效果**：原始九万一千四百三十八字符压缩至四万三千八百三十字符（压缩率百分之四十七点九），四十九分之四十八个样本控制在三千八百字符以内。规则三内置兜底机制，确保即使过滤后无合约保留也不会丢失样本。

## 链上案例

为了验证方法不仅适用于整理后的基准，还能对应真实主网合约，选取了两个公开可验证的链上案例作为未来验证目标：

- **The DAO**：经典重入攻击代表案例，主网地址 `0xbb9bc244d798123fde783fcc1c72d3bb8c189413`，代表函数为 `splitDAO(...)`、`withdrawRewardFor(...)`。攻击路径核心在于资金转出与记账顺序不当，适合验证模型能否识别"外部调用后状态更新"的危险顺序。
- **Lendf.Me**：更接近现代 DeFi 借贷场景，主网地址 `0x0eEe3E3828A45f7601D5F54bF49bB01d1A9dF5ea`，代表接口为 `supply(...)`、`withdraw(...)`、`borrow(...)`。涉及代币回调与跨合约交互，调用链更长、上下文更多，适合检验裁剪与静态摘要是否优于纯原始输入。

## 研究结果

### 消融实验

| 方案 | 准确率 | 误报率 | 漏报率 | 固定变体准确率 |
|---|---|---|---|---|
| baseline_raw | 0.8699 | 0.375 | 0.071 | 0.625 |
| crop_only | **0.9024** | 0.458 | **0.010** | 0.542 |
| crop_slither | 0.8455 | 0.458 | 0.081 | 0.542 |
| crop_slither_cot | 0.8049 | 0.375 | 0.152 | 0.625 |
| crop_slither_multi | 0.8049 | 0.542 | 0.111 | 0.458 |
| 切片 v1（守卫注入） | 0.9024 | 0.375 | 0.030 | 0.625 |
| 裁剪加静态分析加提示词规则 | 0.8780 | 0.292 | 0.081 | 0.708 |
| **切片 v1 加提示词规则** | 0.8699 | **0.250** | 0.101 | **0.750** |

- 准确率最优：crop_only 与切片 v1（守卫注入），均为 0.9024——裁剪是核心增益。
- 误报率最优：切片 v1 加提示词规则（0.250）——固定变体误判大幅减少。
- 漏报率最优：crop_only（0.010）——几乎不漏报。
- 思维链推理与多合约上下文在所有实验中均未带来稳定增益。

### 误报率优化历程

**问题发现**：切片 v1 实验中固定变体被 100% 误判为有漏洞。修饰器固定变体三次运行全误报（置信度 0.90），跨合约固定变体三次运行全误报（置信度 0.95）。

**根因分析**：切片剥离了 ReentrancyGuard 继承链，模型看不到 nonReentrant 修饰器。模型只看到 `call{value:...}` 外部调用模式，直接判定为漏洞。但固定变体已使用 nonReentrant 锁保护，实际不会发生重入。

**三步迭代设计思路**：能否让模型"看到"并"理解"安全保护？守卫源码注入让模型看到，解释性注释让模型理解，提示词规则施加强约束。

| 步骤 | 误报率 | 固定变体准确率 | 核心发现 |
|---|---|---|---|
| 消融基线 | 0.458 | 0.542 | 裁剪丢失 ReentrancyGuard 继承链 |
| 加守卫源码注入 | 0.375 | 0.625 | 模型"看到"修饰器但不理解其语义 |
| 加解释性注释 | 0.375 | 0.625 | **代码内注释完全无效** |
| 加提示词规则 | **0.250** | **0.750** | **系统提示规则最有效** |

**为什么解释性注释无效而提示词规则有效？** 大语言模型训练数据中，代码注释常被模型视为"可忽略的自然语言上下文"而非强约束；系统提示词中的规则被模型视为"指令"，优先级远高于代码内注释。这揭示了重要规律：对大语言模型而言，指令通道不等于内容通道。

**根本局限**：深度求索对话模型对 Solidity 修饰器执行语义（`_` 代表函数体插入位置）理解有上限，即使给出完整源码和规则，仍无法完全正确处理修饰器重入场景。

### 实验迭代的完整逻辑

1. **为什么要消融实验？** 只有逐层剥离才能回答"哪个模块真正有用"——如果直接上最优方案，无法区分收益来源。
2. **为什么要设计切片引擎 v1？** 消融发现 crop_only 已最优但误报率仍高，需要更精准的代码裁剪（全局视角、显著函数、字符上限）。
3. **为什么要守卫注入？** 切片 v1 误报率仍为 0.375，根因是切片剥离了 ReentrancyGuard，必须先让模型"看到"保护锁。
4. **为什么要尝试解释性注释？** 假设模型看到 nonReentrant 源码但不理解，加注释解释 `locked=true` 的语义，结果证明这个假设错了。
5. **为什么要提示词规则？** 既然代码内注释无效，换通道——在系统提示中显式约束——成功。指令通道优于内容通道。

### 分场景指标

| 场景 | 消融最优 | 切片加提示词 | 说明 |
|---|---|---|---|
| 标准重入 | 0.972 | 0.931 | 经典模式几乎完美 |
| 跨函数重入 | 0.833 | 1.000 | 提示词规则后零误报 |
| 修饰器重入 | 0.800 | 0.800 | 修饰器误报顽固 |
| 跨合约重入 | 0.722 | 0.556 | 跨合约最难，跨链样本模式差异大 |

**固定变体误判演变**：消融阶段 baseline_raw 出现九个误报，固定变体准确率 0.625；切片加守卫注入未显著改善；切片加提示词规则后降至六个误报，固定变体准确率升至 0.750。顽固误报来自修饰器固定变体（修饰器执行顺序判断失误）和跨链固定样本（非标准重入模式）。

| baseline_raw 准确率 | 0.935* | **0.870（更真实）** |

*旧数据集准确率因近重复样本而虚高。

旧数据集准确率虚高（近重复样本抬高指标），清洗后更反映真实能力。跨链样本拉低分数（跨合约场景准确率仅 0.611），类别仍然不平衡（33:8），但八个阴性已是当前最佳可达水平。

## 讨论与反思

### 有效的方法（有明确增益）

- **代码裁剪**：去掉无关代码是最大增益，crop_only 始终在准确率前二。核心启示：大语言模型的注意力是稀缺资源，噪声上下文稀释判断。
- **守卫源码注入**：误报率从 0.458 降至 0.375，让模型看到 nonReentrant 修饰器的完整实现。
- **提示词规则教授**：误报率再降至 0.250。核心发现：系统提示中的规则优于代码内注释。
- **数据集清洗**：去重八个近重复样本，样本标识前缀化消除碰撞，误报率评估更可靠。

### 无效或退化（投入无回报）

- **思维链推理与多合约上下文**：在所有实验中均未带来稳定提升。复杂提示不等于更好结果，有时反而引入噪声。
- **解释性注释**：在代码注释中添加"上锁操作"的说明对误报率无任何改善。模型把代码注释当作文档而非指令。
- **跨链数据源**：样本模式差异大（跨链桥调用而非典型重入），拉低整体准确率。

### 关键洞察：指令通道 vs 内容通道

大语言模型处理输入时有两条信息通道：指令通道（系统提示词中的规则，优先级高）和内容通道（代码和注释，优先级低）。本实验首次在重入检测场景下验证了这一机制。

### 设计经验教训

1. 先清洗数据再实验：旧数据集有近重复样本导致准确率虚高约六个百分点，清洗后才暴露真实能力。数据质量检查应是最优先步骤。
2. 提示词规则应第一步就加入：从第一轮消融就应该教授 nonReentrant 语义。对大语言模型的知识盲区要有预判。
3. 代码注释不可靠：不应假设模型会"认真读注释"——模型对自然语言注释的敏感度远低于结构化指令。
4. 消融实验设计要严格：每次只变一个变量，否则无法归因。五个方案从原始到裁剪到静态分析到思维链到多合约的递进式设计是成功的。
5. 跨链样本不应纳入基准：其漏洞模式与典型重入不同（跨链桥 vs 资金重入），应作为独立任务评估。

### 局限性与威胁有效性

- 类别不平衡：三十三正比八负，误报率分母仅八，单次误判仍可影响结果。
- 模型能力上限：深度求索对话模型无法完全理解 nonReentrant 修饰器的执行语义。
- 公开数据泄漏风险：智慧漏洞精选集合约均为以太坊浏览器已验证的主网合约，预训练可能包含。
- 样本规模有限：四十一个样本适合消融分析，但距离大规模基准仍有距离。
- 零样本设置：未进行微调或少样本示例引导，所有实验均在零样本条件下完成。

### 未来工作

- 更强的模型：尝试 GPT-4o 等模型，验证修饰器语义理解能力。
- 链上合约验证：将 The DAO、Lendf.Me 纳入流水线。
- 自洽性投票机制：用多次采样加固低置信度样本的判定稳定性。
- 数据集扩展：寻找更多配对的已修复与不安全样本（当前仅八对）。
- 检索增强生成方案探索：参考论文建议的方向。

## 结论

本研究搭建了完整的可复现重入漏洞检测流水线（消融→切片→守卫→提示词四层优化）。

**消融实验**：crop_only 在准确率上最优（0.9024），代码裁剪是核心增益模块。

**切片优化**：切片引擎 v1 通过四项规则压缩百分之四十七点九，结合守卫注入和提示词规则实现误报率从 0.458 降至 0.250（降幅百分之四十五）。

**误报率三步优化**：守卫注入 → 解释性注释（无效）→ 提示词规则（有效），验证了大语言模型对系统提示规则比代码注释更敏感。

**最终方案**：切片 v1 加提示词规则，准确率 0.870，误报率 0.250，固定变体准确率 0.750。

核心价值：不仅是"模型判断是否"，更是将漏洞分析过程结构化、可记录、可复核，并通过消融实验精准定位每个改进模块的增益来源。
